# 污水厂 PINN + RTO 修正后完整技术实现路线

## 总体架构：四层分离式 RTO 系统

```
┌─────────────────────────────────────────────────────────────────┐
│  Layer 0: PINN 参数校准器（离线，每日/每周）                       │
│  角色：逆问题求解器，从运行数据反推 ASM1 动力学参数                  │
│  输出：校准后的参数 θ = {μ_H, K_S, μ_A, K_NH, ...}              │
└────────────────────────────┬────────────────────────────────────┘
                             │ 参数 θ
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│  Layer 1: 混合动态预测模型（在线，每 15-30 分钟）                  │
│  结构：dx/dt = f_ASM1(x, u; θ) + g_NN(x, u; φ)                │
│  角色：状态预测引擎，预测未来 1-2h 的水质演化轨迹                   │
└────────────────────────────┬────────────────────────────────────┘
                             │ 预测轨迹 x̂(t)
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│  Layer 2: 成本优化器（在线，每 15-30 分钟）                       │
│  目标函数：minimize C_elec(u) + C_chem(u)（WaterTAP OPEX）      │
│  约束：出水 S_NH, TN, COD ≤ 排放标准（含安全裕量）                │
│  输出：最优控制指令 u*                                           │
└────────────────────────────┬────────────────────────────────────┘
                             │ u*
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│  Layer 3: 反馈校正 + 安全机制                                    │
│  每周期用滤波后实测值修正预测起点；异常时退回保守策略                 │
│  输出：校正后的控制指令 → 下发 PLC/DCS                            │
└─────────────────────────────────────────────────────────────────┘
```

## Layer 0：PINN 参数校准器

### 定位

PINN 不做预测，只做它最擅长的事——**逆问题求解**。从历史运行数据中反推 ASM1 动力学参数，使其跟随工况变化自适应更新。

### 输入

- 过去 7-30 天的运行数据（进水水质、出水水质、操作参数、水温等）
- ASM1 核心方程结构（R1 好氧异养、R2 反硝化、R3 硝化）

### 输出

- 校准后的关键动力学参数（初期 2-3 个，逐步增加，详见参数可辨识性策略）
- 每个参数带合理范围约束（基于 ASM1 文献值上下限）

### 训练损失

$$L_{total} = L_{data} + \alpha \cdot L_{physics} + \beta \cdot L_{reg} + \lambda \cdot L_{smooth}$$

| 损失项               | 权重建议    | 作用           |
| ----------------- | ------- | ------------ |
| L_data（数据拟合）      | 1.0     | 预测精度         |
| L_physics（ODE 残差） | 0.1     | 物理一致性        |
| L_reg（参数范围正则）     | 0.01    | 防止参数越界       |
| L_smooth（时间平滑）    | 0.1-1.0 | 防止参数在相邻周期间突跳 |

### 校准频率

- **常规**：每天用最近 7 天数据重新校准
- **触发式**：当 Layer 1 预测偏差连续超过阈值时，立即触发重校准

### 关键设计要点

- 参数网络输入包含工况特征（水温、SRT、MLSS、平均负荷），使参数能随工况自适应
- 输出层用 Softplus 激活保证参数为正，再乘以缩放因子限制在文献合理范围

### 参数可辨识性策略（防止过度参数化）

同时放开 9 个参数进行校准存在严重的**参数不可辨识性**风险——多组参数组合可能产生几乎相同的 Loss，但物理意义完全不同（如出水氨氮偏高，可能是 μ_A 变小，也可能是 K_NH 变大）。

**应对策略：**

**分层放开，逐步增加自由度：**

| 阶段  | 可学习参数                | 固定参数         | 依据               |
| --- | -------------------- | ------------ | ---------------- |
| 初期  | μ_H, μ_A（2 个）        | 其余 7 个锁定为文献值 | 最大生长速率对温度和污泥龄最敏感 |
| 中期  | + K_S, K_NH（4 个）     | 其余 5 个       | 半饱和系数影响动态响应形状    |
| 成熟期 | + b_H, b_A, η_g（7 个） | K_OH, K_OA   | 衰减系数和缺氧校正因子      |

> 建议在每个阶段上线前，用 **Morris 全局敏感性分析** 确认哪些参数对出水指标影响最大，优先放开高敏感参数。

**参数时间平滑惩罚：**

水厂的菌群演替是缓慢的（天到周级别），参数不应在相邻校准周期间剧烈跳变。在校准损失中加入：

$$L_{smooth} = \lambda \sum_i \| \theta_i^{(today)} - \theta_i^{(yesterday)} \|^2$$

其中 λ 为平滑系数（建议初始值 0.1-1.0，根据参数稳定性调整）。这确保了：昨天 μ_A = 0.5，今天不会突跳到 0.8。

**<mark>需要预测的9个参数</mark>**

### Layer 0 (PINN) 预测/校准出来的参数有哪些？

Layer 0 的核心任务是做“系统辨识”。它要从水厂的历史表格中，把那些看不见、摸不着的**微生物动力学参数 ($\theta$)** 反推出来。根据前文的简化 ASM1 方案，它最多需要预测以下 9 个参数：

**核心生长与衰减类：**

1. $\mu_H$ (最大异养生长速率)：决定了去除 COD 和消耗溶解氧的基础速度。

2. $\mu_A$ (最大自养生长速率)：决定了硝化反应（除氨氮）的速度，对温度极度敏感。

3. $b_H$ (异养衰减系数)：异养菌的死亡率。

4. $b_A$ (自养衰减系数)：硝化菌的死亡率。

**亲和力（半饱和）类：**

5. $K_S$ (基质半饱和系数)：异养菌对有机物 (COD) 的亲和力。

6. $K_{NH}$ (氨半饱和系数)：硝化菌对氨氮的亲和力。

7. $K_{OH}$ (异养好氧半饱和系数)：异养菌对氧气的争夺能力。

8. $K_{OA}$ (自养好氧半饱和系数)：自养菌对氧气的争夺能力。

**特殊校正项：**

9. $\eta_g$ (反硝化缺氧校正因子)：在缺氧环境下，异养菌用硝酸盐呼吸时的生长折减系数。

<mark>**预测9个参数所需的数据类别**</mark>

### 预测这 9 个参数，到底需要喂给网络哪些信息？

PINN 作为“逆问题求解器”，它的逻辑是：**已知系统的输入和输出轨迹，去猜系统内部的物理“齿轮”是怎么咬合的。** 因此，你需要给它提供一个完整时间窗口（比如过去 7 到 30 天）内的历史运行时序数据。

以下是预测这 9 个参数所**必须完整输入**的特征变量清单（包含英文全称与通用简写/符号），绝无省略。这些特征在工程上分为五大类：

### 一、 进水水质与环境扰动特征 (Influent & Environmental Disturbances)

这是微生物的“食物来源”和“生存环境”，直接决定了最大生长速率（$\mu_H, \mu_A$）和亲和力系数（$K_S, K_{NH}$）的激发程度。

1. **进水流量** * 英文全称: Influent Flow Rate
   
   - 简写/符号: $Q_{in}$ (或 $Q$)

2. **水温** (极度关键，所有 $\mu$ 和 $b$ 参数都随温度呈指数变化)
   
   - 英文全称: Temperature
   
   - 简写/符号: $T$

3. **进水易生物降解有机物 / 进水可溶性 COD** (异养菌的直接食物)
   
   - 英文全称: Influent Readily Biodegradable Substrate / Influent Soluble COD
   
   - 简写/符号: $S_{S,in}$ (或 $S_{COD,in}$)

4. **进水氨氮** (自养硝化菌的直接食物)
   
   - 英文全称: Influent Ammonium Nitrogen
   
   - 简写/符号: $S_{NH,in}$ (或 $NH_3\text{-}N_{in}$)

5. **进水总氮** (包含有机氮，用于核算总氮负荷)
   
   - 英文全称: Influent Total Nitrogen
   
   - 简写/符号: $TN_{in}$

6. **进水碱度** (硝化反应的缓冲剂，影响 $K_{NH}$ 和反应速率)
   
   - 英文全称: Influent Alkalinity
   
   - 简写/符号: $S_{ALK,in}$ (或 $ALK_{in}$)

7. **进水硝酸盐氮** (通常较低，但属于完整物料衡算的一部分)
   
   - 英文全称: Influent Nitrate Nitrogen
   
   - 简写/符号: $S_{NO,in}$

### 二、 操作与工艺控制特征 (Operational & Control Actions)

这是水厂施加的“人为干预”，决定了氧气供给和物质停留时间，用于推算氧亲和力（$K_{OH}, K_{OA}$）和衰减率（$b_H, b_A$）。

8. **氧传质系数 / 曝气速率** (风机变频器反馈)
   
   - 英文全称: Oxygen Transfer Coefficient / Aeration Rate
   
   - 简写/符号: $KLa$ (或 $Q_{air}$)

9. **内回流流量 / 内回流比** (将硝态氮送回缺氧区，用于测算反硝化校正因子 $\eta_g$)
   
   - 英文全称: Internal Recycle Flow Rate / Internal Recycle Ratio
   
   - 简写/符号: $Q_{IR}$ (或 $R$)

10. **外回流流量 / 污泥回流比** (维持反应器内的生物量)
    
    - 英文全称: Return Activated Sludge Flow Rate / RAS Ratio
    
    - 简写/符号: $Q_{RAS}$ (或 $r$)

11. **剩余污泥排放流量** (排泥量，决定微生物群落新陈代谢快慢的根本阀门)
    
    - 英文全称: Waste Activated Sludge Flow Rate
    
    - 简写/符号: $Q_{WAS}$ (或 $Q_W$)

12. **外部碳源投加量** (若有，补充 $S_S$)
    
    - 英文全称: External Carbon Dosing Rate
    
    - 简写/符号: $D_{carbon}$

13. **化学药剂投加量** (如除磷药剂或补碱，影响 $S_{ALK}$)
    
    - 英文全称: Chemical Dosing Rate
    
    - 简写/符号: $D_{chem}$

### 三、 反应器内部状态特征 (Reactor Internal States)

这是生化反应发生的“现场实况”，模型需要用这些数据来校准它对物理主干公式的计算。

14. **溶解氧浓度** (好氧池各段实测值，推算 $K_{OH}, K_{OA}$ 必不可少)
    
    - 英文全称: Dissolved Oxygen Concentration
    
    - 简写/符号: $S_O$ (或 DO)

15. **混合液悬浮固体浓度** (用来近似折算异养活性生物 $X_{BH}$ 和自养活性生物 $X_{BA}$ 的基数)
    
    - 英文全称: Mixed Liquor Suspended Solids
    
    - 简写/符号: MLSS

16. **混合液挥发性悬浮固体浓度** (更精确的有效生物量表征)
    
    - 英文全称: Mixed Liquor Volatile Suspended Solids
    
    - 简写/符号: MLVSS

17. **反应器有效容积** (常数特征，用于计算浓度变化率)
    
    - 英文全称: Reactor Volume
    
    - 简写/符号: $V$

### 四、 出水水质特征 (Effluent Observations / Targets)

这是 PINN 计算 Loss（损失函数）的“真实标签”。参数猜得对不对，全靠和这些出水指标对答案。

18. **出水氨氮** * 英文全称: Effluent Ammonium Nitrogen
    
    - 简写/符号: $S_{NH,out}$ (或 $NH_3\text{-}N_{e}$)

19. **出水硝酸盐氮** * 英文全称: Effluent Nitrate Nitrogen
    
    - 简写/符号: $S_{NO,out}$ (或 $NO_3\text{-}N_{e}$)

20. **出水总氮** * 英文全称: Effluent Total Nitrogen
    
    - 简写/符号: $TN_{out}$ (或 $TN_e$)

21. **出水可溶性 COD / 出水易降解有机物** * 英文全称: Effluent Soluble COD
    
    - 简写/符号: $S_{S,out}$ (或 $COD_e$)

22. **出水碱度** * 英文全称: Effluent Alkalinity
    
    - 简写/符号: $S_{ALK,out}$

### 五、 关键衍生特征 (Critical Derived Features)

在算法工程中，直接把原始流量和浓度喂给网络效果往往不好。通常需要用上述原始特征计算出以下两个宏观衍生特征喂给网络，这能极大加速 PINN 的收敛，因为它们直接反映了系统的停留时间和物理压力。

23. **水力停留时间** (水在池子里的时间，等于 $V / Q_{in}$)
    
    - 英文全称: Hydraulic Retention Time
    
    - 简写/符号: HRT

24. **污泥龄 / 固体停留时间** (极其关键：异养菌和自养菌的衰减率 $b_H, b_A$ 完全依赖于这个指标)
    
    - 英文全称: Sludge Retention Time / Solid Retention Time
    
    - 简写/符号: SRT

**逻辑闭环**：网络拿着【干扰变量】和【控制变量】，代入它自己当前猜想的 $\theta$（那 9 个参数），通过 ODE 积分向前推演，算出一条模拟的【响应状态】轨迹。然后把模拟轨迹和真实的【响应状态】做对比算 Loss，再通过反向传播去修正它猜想的 $\theta$。

**<mark>如何分批预测9个参数</mark>**

#### 阶段 1：全局锁定，只放开“主引擎”速度 (初期)

- **动作**：在网络设计时，将除了最大异养生长速率 $\mu_H$ 和最大自养生长速率 $\mu_A$ 之外的其余 7 个参数，全部**硬编码为 WaterTAP 或 ASM1 里的文献默认常数**。只让 PyTorch 将 $\mu_H$ 和 $\mu_A$ 声明为可求导的张量（Tensors）。

- **为什么**：这两个参数决定了系统消耗 COD 和氨氮的“最大潜力”，对水温和污泥龄的变化最为敏感。此时系统自由度极低，优化器只能乖乖地去寻找最合理的生长速率，绝不会出现“用变异的亲和力来凑结果”的现象。

#### 阶段 2：引入“敏感度分析”，解锁曲线形状 (中期)

- **动作**：当 $\mu_H$ 和 $\mu_A$ 的预测轨迹在几个周期内稳定下来后，你可以开始解锁半饱和系数 $K_S$（基质亲和力）和 $K_{NH}$（氨氮亲和力）。

- **关键前置操作**：在解锁前，必须运行一次 **Morris 全局敏感性分析**。这是一种数学工具，用于评估：如果我微微改变 $K_S$，出水 COD 会发生明显变化吗？如果变化极小，说明当前的工况数据根本不足以支撑网络学习到真实的 $K_S$，那就必须继续把它锁死；如果变化明显，则可以将其加入可学习的张量列表中。

#### 阶段 3：完全体解锁与时间平滑 (成熟期)

- **动作**：最后阶段，把死亡率（$b_H, b_A$）和缺氧校正因子（$\eta_g$）也变为可学习参数。此时 9 个参数全部由网络动态输出。

- **必须配套的护栏**：此时极易发生参数崩溃，必须在 Loss 函数中加入**参数时间平滑惩罚 ($L_{smooth}$)**。物理常识告诉我们，水厂的菌群演替是缓慢的。如果网络昨天算出 $\mu_A = 0.5$，今天算出 $\mu_A = 0.8$，这在生物学上是不可能的。平滑惩罚会强制要求今天的参数值必须在昨天参数值的附近小幅微调，从而彻底按死参数在多维空间中乱跑的可能。

总结来说，分批预测不是指把数据切开，而是**在代码的架构层面上，随着模型对系统理解的深入，像解开封印一样，把原本写死的标量常数（Constants），一个个替换为网络可以梯度更新的权重（Weights）。**

## Layer 1：混合动态预测模型

### 定位

系统的**核心预测引擎**。不是静态映射，而是沿时间积分的动态模型。

<mark>**“动态”的频率：** 这里有两个层面的频率需要区分：</mark>

1. **执行预测的频率（在线调用频率）**：通常是**每 15-30 分钟一次**（跟你的 RTO 优化周期一致）。即每 15 分钟，系统拉取当前传感器数据，向未来推演一次。

2. **ODE 内部的积分步长（微观时间分辨率）**：在预测未来 2 小时的轨迹时，积分求解器内部可能是在以**秒级或分钟级**的极小步长在滚动计算，以此保证刚性微分方程的数值稳定性。

### 模型结构

```
dx/dt = f_ASM1(x, u; θ) + g_NN(x, u; φ)
        ├── 物理主干           ├── 残差补偿网络
        ├── 保证方向正确        ├── 补偿简化偏差
        └── 使用 Layer 0 参数   └── 规模小，L2 正则化
```

### 状态变量与控制变量

**核心状态变量（6 个）：**

| 变量    | 含义    | 单位     | 与优化目标的关系  |
| ----- | ----- | ------ | --------- |
| S_S   | 易降解基质 | mg/L   | 影响 COD 出水 |
| S_O   | 溶解氧   | mg/L   | 曝气控制目标    |
| S_NH  | 氨氮    | mg/L   | 出水达标核心    |
| S_NO  | 硝态氮   | mg/L   | 总氮达标核心    |
| X_BH  | 异养菌浓度 | mg/L   | 生物量       |
| S_ALK | 碱度    | mmol/L | 加药控制目标    |

**控制变量：**

| 变量     | 含义    | 单位   | 可控性        |
| ------ | ----- | ---- | ---------- |
| KLa    | 氧传质系数 | 1/h  | 对应风机频率，可控  |
| Q_r    | 回流比   | 无量纲  | 对应回流泵，可控   |
| D_chem | 加药量   | mg/L | 对应加药泵，可控   |
| Q_in   | 进水流量  | m³/h | 不可控，但需作为输入 |

### 物理主干 f_ASM1

完整的物料衡算 ODE，每个状态变量的变化率 = 水力输运项 + 物理过程项 + 生化反应项：

| 状态变量       | 水力输运                   | 物理/化学过程                  | 关键反应                |
| ---------- | ---------------------- | ------------------------ | ------------------- |
| S_S（易降解基质） | (S_S_in - S_S) / HRT   | —                        | R1 消耗、R2 消耗、R7 水解生成 |
| S_O（溶解氧）   | -S_O / HRT             | +KLa·(S_O_eq - S_O) 曝气传质 | R1 耗氧、R3 耗氧（最大）     |
| S_NH（氨氮）   | (S_NH_in - S_NH) / HRT | —                        | R3 硝化消耗             |
| S_NO（硝态氮）  | -S_NO / HRT            | —                        | R3 硝化生成、R2 反硝化消耗    |
| X_BH（异养菌）  | —                      | —                        | R1/R2 生长、b_H 衰减     |
| S_ALK（碱度）  | -S_ALK / HRT           | +加药补碱项                   | R3 消耗碱度、R2 回收碱度     |

**核心反应速率方程（3 个）：**

**R1 好氧异养生长**（主要耗氧 + COD 去除）：

$$\rho_1 = \mu_H \cdot \frac{S_S}{K_S + S_S} \cdot \frac{S_O}{K_{OH} + S_O} \cdot X_{BH}$$

**R3 硝化**（最大耗氧 + 氨氮去除 + 消耗碱度）：

$$\rho_3 = \mu_A \cdot \frac{S_{NH}}{K_{NH} + S_{NH}} \cdot \frac{S_O}{K_{OA} + S_O} \cdot X_{BA}$$

**R2 反硝化**（消耗硝态氮 + 回收碱度 + 不耗氧）：

$$\rho_2 = \mu_H \cdot \frac{S_S}{K_S + S_S} \cdot \frac{K_{OH}}{K_{OH} + S_O} \cdot \frac{S_{NO}}{K_{NO} + S_{NO}} \cdot \eta_g \cdot X_{BH}$$

**<mark>f_ASM1所用到的参数</mark>**

### $f_{ASM1}(x, u; \theta)$ 所用到的参数具体有哪些？

这个模块是纯纯的物理学霸，它严格按照你提取的 ASM1 简化公式计算理论反应速率。

- **$x$ (状态变量 - State)**：水池里当前的实际情况。包括 $S_S$ (易降解 COD)、$S_O$ (溶解氧)、$S_{NH}$ (氨氮)、$S_{NO}$ (硝态氮)、$X_{BH}$ (异养菌浓度)、$S_{ALK}$ (碱度)。

- **$u$ (控制变量 - Action)**：你或者优化器下达的操作指令。包括 $KLa$ (曝气风机对应的氧传质系数)、$Q_r$ (回流泵的回流比)、$D_{chem}$ (加药泵的加药量) 以及外部不可控但必须输入的进水流量 $Q_{in}$。

- **$\theta$ (动力学参数 - Weights)**：就是 Layer 0 刚刚校准完传过来的那一组参数（<mark>预测出来的那9个参数</mark>，比如 $\mu_H, \mu_A$）。

### 残差补偿网络 g_NN

- **作用**：学习 ASM1 简化后丢失的动态特征（水解过程 R7、温度非线性效应、进水冲击响应等）
- **设计原则**：
  - 网络规模要小（2-3 层，每层 32-64 神经元），避免"喧宾夺主"
  - 强 L2 正则化，鼓励残差趋近于零——只有当物理主干确实解释不了的偏差时才激活
  - 输入包含历史窗口（见"时间延迟的显式建模"），输出为 6 维修正向量
  - 残差输出限幅（如 ±20% 的物理主干幅值），防止残差网络主导行为

### 时间积分方法

ASM1 方程组是**刚性系统**（stiff system）：DO 响应秒级，硝化小时级，污泥天级。

| 阶段   | 推荐求解器                                       | 原因         |
| ---- | ------------------------------------------- | ---------- |
| 训练阶段 | `torchdiffeq` 隐式求解器                         | 支持反向传播计算梯度 |
| 推理阶段 | `scipy.integrate.solve_ivp(method='Radau')` | 更快更稳定      |

> **禁止使用简单 Euler 法** —— 步长稍大就会数值发散。必须用自适应步长的隐式方法（Radau IIA 或 BDF）。

### 训练方法

- **训练数据**：历史运行时序数据，按滑动窗口切成"初始状态 + 未来轨迹"的训练样本

- **损失函数**：
  
  - 轨迹预测损失：预测轨迹 vs 实测轨迹的 MSE
  - 物理一致性正则化：ODE 残差（增强鲁棒性）
  - 残差网络 L2 正则化

- **训练频率**：离线训练；每周/每月用最新数据微调（增量学习）

- **训练数据集**：<mark>与训练PINN的数据相同

- **训练数据集的使用方法**：### 用法的区别：切分与视角的差异
  
  - **给 PINN (Layer 0) 用：大时间窗口看整体（找物理常数）**
    
    - **切分方式**：通常切成比较长的时间块，比如连续 7 天或 14 天的完整数据作为一段。
    
    - **视角**：PINN 要看这几天内水厂整体的物质平衡和能量流动，从而反推出这 7 天里微生物的平均“性格”（比如最大生长速率 $\mu_H$、$\mu_A$）。它关注的是宏观的物理平衡，去解出那些恒定的动力学参数。
  
  - **给 $g_{NN}$ (Layer 1) 用：滑动窗口看细节（做动态推演）**
    
    - **切分方式**：把同一批数据，用<mark>滑动时间窗口（Sliding Window）</mark>切成无数个短促的“前因后果片段”（例如：以 [10:00-10:15 的状态和控制动作] 为输入，以 [10:15-12:00 的轨迹] 为标签）。
    
    - **视角**：$g_{NN}$ 就像一个盯盘的操作员。它需要从这些切碎的短片段中，学习微观的非线性动态特征（比如：当风机频率突然拉满时，系统存在 10 分钟的物理延迟，这种物理公式 $f_{ASM1}$ 算不出来的细节滞后，全靠 $g_{NN}$ 强行记住并补偿）。

**<mark>残差补偿网络 g_NN的作用</mark>**

### $g_{NN}(x, u; \phi)$ 学习到丢失的特征后要干什么？

这是最体现算法工程巧思的地方。

你把庞大的 ASM1 简化成了只有 3 个宏观方程，这意味着你不可避免地丢掉了很多细节（比如大分子有机物的水解过程、水温的非线性影响、或者是某种未知干扰）。如果不补上这些，物理主干算出来的变化率 $\frac{dx_{physics}}{dt}$ 总是和现实差那么一点点。

$g_{NN}$ 是一个轻量级的多层感知机 (MLP)。它的输入同样是当前状态 $x$ 和控制指令 $u$，它的网络权重是 $\phi$。

它学习完丢失的特征后，**唯一的任务就是输出一个“残差向量” $\frac{dx_{residual}}{dt}$，并直接加到物理主干的计算结果上。**

最终在 ODE 求解器里滚动的那个完整的导数是：

$$\frac{dx_{total}}{dt} = f_{ASM1}(x, u; \theta) + g_{NN}(x, u; \phi)$$

**举个通俗的例子**：

物理主干 $f_{ASM1}$ 算出来：当前条件下，氨氮每小时应该下降 1.0 mg/L。

但真实水厂因为某种复杂的管网发酵原因（被简化掉了），实际每小时只下降了 0.8 mg/L。

此时，$g_{NN}$ 经过历史数据的训练，在这个特定的工况下“察觉”到了这种非物理的阻力，于是它输出一个 $+0.2$ 的修正项。

两者一相加：$-1.0 + 0.2 = -0.8$。

模型不仅完美契合了实际，而且依然保持了“曝气越多、氨氮降得越快”的物理大方向

### 多反应器串联

如果曝气池存在推流特征，用 **N 个 CSTR 串联** 近似：

```
CSTR₁(缺氧) → CSTR₂(缺氧) → CSTR₃(好氧) → CSTR₄(好氧) → CSTR₅(好氧)
                                              ↑
                                        回流 ←─── 二沉池出水
```

每个 CSTR 有独立的 6 个状态变量，总系统维度 = 6 × N。下游 CSTR 的进水 = 上游 CSTR 的出水。

### 时间延迟的显式建模

真实污水厂存在显著的**纯时间延迟**（Dead Time）：进水水质突变可能需要数小时才能传输到好氧区末端被传感器捕捉。CSTR 串联模型能模拟扩散混合，但对纯滞后的刻画不够精确。如果预测模型的时间相位对不上，优化器将在错误的时间下达控制指令。

**应对策略（二选一或组合使用）：**

**方案 A：显式时间平移队列（推荐，简单可靠）**

在状态更新前加入延迟缓冲区：

```
实际进水 x_in(t) → [延迟队列, τ = V/Q] → 模型输入 x_in(t - τ)
```

- 延迟时间 τ 根据各反应器的水力停留时间（HRT）设定
- 不同反应器的延迟不同（缺氧区 → 好氧区 → 二沉池，累计可达 6-12 小时）

**方案 B：残差网络加入历史窗口**

将残差补偿网络 g_NN 的输入从当前状态扩展为包含历史窗口：

```
g_NN 输入: [x(t), x(t-1), x(t-2), ..., x(t-k), u(t)]
```

- 历史窗口长度 k = 3-6 个时步（覆盖 1-3 小时）

- 这赋予残差网络"记忆能力"，使其能学习到延迟效应

- 实现简单：只需将输入维度从 (6+4) 扩展为 (6×(k+1)+4)
  
  **建议**：初期用方案 A（确定性强，可解释），方案 B 作为补充提升精度。

## Layer 2：成本优化器

### 定位

基于 Layer 1 的预测能力，寻找使运营成本最低且出水达标的最优控制策略。

### 目标函数（从 WaterTAP 提取，仅 OPEX）

$$\text{minimize:} \quad J(u) = C_{elec}(u) + C_{chem}(u)$$

| 成本项    | 计算公式                                                                    | 关键参数                          |
| ------ | ----------------------------------------------------------------------- | ----------------------------- |
| 曝气电力成本 | $C_{elec} = P_{elec} \times E_{aeration} \times \Delta t$               | P_elec: 本地电价 (元/kWh)          |
|        | $E_{aeration} = \frac{S_{O,eq}}{1.8} \times V \times KLa / 1000$ [kW]   | 来自 WaterTAP cstr_injection.py |
| 加药成本   | $C_{chem} = D_{chem} \times Q \times \Delta t \times C_{unit} / purity$ | C_unit: 本地药剂单价 (元/kg)         |

> **关键修改**：将 WaterTAP 默认的美国经济参数替换为中国本地值（电价、药价）。

### 约束条件

| 约束      | 排放标准（一级 A） | 优化目标（含 20% 安全裕量） |
| ------- | ---------- | ---------------- |
| 出水 S_NH | ≤ 5 mg/L   | ≤ 4 mg/L         |
| 出水 TN   | ≤ 15 mg/L  | ≤ 12 mg/L        |
| 出水 COD  | ≤ 50 mg/L  | ≤ 40 mg/L        |
| DO 范围   | —          | 0.5 - 4.0 mg/L   |
| KLa 范围  | —          | 设备物理限制           |

### 控制变量

| 变量               | 含义         | 对应执行机构 |
| ---------------- | ---------- | ------ |
| KLa₃, KLa₄, KLa₅ | 各好氧池氧传质系数  | 风机变频器  |
| Q_r              | 内回流比       | 回流泵变频器 |
| D_chem           | 加药量 (mg/L) | 加药泵    |

### 优化求解方法

| 方法                  | 适用性     | 单次耗时     | 推荐程度     |
| ------------------- | ------- | -------- | -------- |
| 基于梯度优化（Adam/L-BFGS） | 端到端可微链条 | < 1 秒    | ⭐⭐⭐⭐⭐ 首选 |
| SQP（序列二次规划）         | 经典约束优化  | 1-5 秒    | ⭐⭐⭐⭐ 备选  |
| 遗传算法/粒子群            | 无梯度方法   | 10-60 秒  | ⭐⭐ 不推荐   |
| Pyomo + IPOPT       | 传统方式    | 30秒-10分钟 | ⭐ 不推荐    |

**约束处理**：安全裕量法（收紧排放标准 20%）+ 罚函数法（超标部分加大权重惩罚）。

### 优化输出示例

```
u* = argmin J(u)  s.t. 出水达标

输出：
- KLa₃ = 8.5 /h  → 转换为风机频率 → 下发 PLC
- KLa₄ = 6.2 /h
- KLa₅ = 4.0 /h
- Q_r = 1.2（回流比）
- D_chem = 15 mg/L NaOH
```

## Layer 3：反馈校正 + 安全机制

### 定位

弥合模型残余偏差，保障工程安全。这是整个系统能否在实际污水厂运行的**关键层**。

### 3.1 反馈校正（含传感器噪声防护）

每个 RTO 周期开始时，用最新实测值修正预测模型的初始状态：

$$\hat{x}(t_0) = x_{filtered}(t_0) \quad \leftarrow \text{注意：必须使用滤波后的值，而非原始实测值}$$

#### 传感器噪声与延迟问题

> ⚠️ **绝对不能用原始的 x_measured 直接重置预测起点。**

污水厂在线仪表（氨氮仪、总氮仪等）普遍存在以下问题：

| 问题   | 表现          | 对 RTO 的影响    |
| ---- | ----------- | ------------ |
| 测量延迟 | 15-30 分钟滞后  | 预测起点与真实状态不同步 |
| 清洗峰值 | 自清洗时产生假数据脉冲 | 导致预测轨迹剧烈抖动   |
| 探头漂移 | 数值缓慢偏移      | 引入系统性偏差      |
| 随机噪声 | ±5-15% 波动   | 控制指令频繁震荡     |

如果直接用含噪声/延迟的原始数据硬重置 ODE 起点，会导致**风机和加药泵频繁启停**，不仅不节能反而加速设备磨损。

#### 必须加入的数据预处理层

```
原始实测值 → [异常值检测] → [清洗峰值剔除] → [平滑滤波] → 滤波后的值用于重置
```

**平滑滤波方法选择：**

| 方法          | 复杂度  | 效果             | 推荐阶段      |
| ----------- | ---- | -------------- | --------- |
| 指数移动平均（EMA） | ★☆☆☆ | 简单有效，α=0.2-0.3 | 初期首选      |
| 中值滤波        | ★☆☆☆ | 对脉冲噪声鲁棒        | 配合 EMA 使用 |
| 一维卡尔曼滤波     | ★★☆☆ | 最优线性估计         | 中期升级      |
| 移动视界估计（MHE） | ★★★★ | 约束下最优          | 成熟期       |

> **建议**：初期用 **中值滤波（去脉冲）+ EMA（平滑）** 的两级组合，已能覆盖绝大部分传感器问题。

#### 进阶反馈校正方法

| 方法                  | 复杂度  | 效果           |
| ------------------- | ---- | ------------ |
| 滤波后状态重置             | ★☆☆☆ | 消除累积漂移（推荐初期） |
| 输出偏差校正（预测值 + 线性偏差项） | ★★☆☆ | 补偿系统性偏差      |
| 扩展卡尔曼滤波（EKF）        | ★★★☆ | 最优状态估计       |
| 移动视界估计（MHE）         | ★★★★ | 约束下的最优估计     |

### 3.2 安全机制

| 机制          | 触发条件                           | 动作                       |
| ----------- | ------------------------------ | ------------------------ |
| **预测置信度监测** | 输入工况超出训练数据分布（Mahalanobis 距离检测） | 输出告警，切换到保守模式             |
| **控制增量限幅**  | 单次优化的控制变量变化幅度过大                | 限制每次调整 ≤ 10-15%          |
| **出水超标预警**  | 预测出水浓度接近标准值 90%                | 自动加大安全裕量                 |
| **回退机制**    | 模型预测偏差连续 3 个周期 > 阈值            | 退回人工设定的固定控制策略            |
| **硬约束保护**   | 任何情况下                          | DO ≥ 0.5 mg/L；加药量 ≤ 设计上限 |

### 3.3 未测量干扰的异常追踪

实际水厂中，部分关键干扰无法实时测量（进水 BOD/COD 化验滞后数小时、重金属突入致菌群中毒等）。当这类未测量干扰发生时，Layer 1 的物理主干会完全失效——模型还在预测"一切正常"，但实际系统已经崩溃。

**核心哨兵信号：DO 异常升高**

当发生菌群中毒/抑制时，微生物停止耗氧，而曝气系统仍在运行，导致 DO 异常升高。这是最灵敏的间接检测信号：

```
正常：DO ≈ 1.5-3.0 mg/L（被微生物消耗维持平衡）
异常：DO > 4.5 mg/L 且持续上升（微生物活性骤降）
```

**异常追踪器设计：**

| 监测指标           | 正常范围       | 一级告警         | 二级告警（切断 AI 控制） |
| -------------- | ---------- | ------------ | -------------- |
| DO 偏差（实测 - 预测） | < 0.5 mg/L | 0.5-1.5 mg/L | > 1.5 mg/L     |
| 出水 NH₃-N 偏差    | < 1.0 mg/L | 1.0-2.0 mg/L | > 2.0 mg/L     |
| 连续偏差周期数        | 0-1        | 2-3          | ≥ 4（连续 1 小时）   |

**响应动作：**

| 告警级别 | 自动动作                                   |
| ---- | -------------------------------------- |
| 一级告警 | 加大安全裕量至 30%；缩短优化周期至 10 分钟；通知运维人员       |
| 二级告警 | **立即切断 AI 自动控制**；回退到人工预设的保守固定策略；触发紧急通知 |

> **原则**：对于无法被模型预见的黑天鹅事件，唯一可靠的防线是"快速发现 + 立即退出"，而不是试图让模型去适应它。

## 核心创新点

1. **PINN 做参数校准而非预测**：精准定位 PINN 的能力边界，让它做最擅长的逆问题求解
2. **分层参数放开策略**：从 2 个参数起步，逐步增加自由度，避免不可辨识性陷阱
3. **物理主干 + 残差补偿的混合结构**：既保证物理因果正确性（增加曝气 → 氨氮下降），又有数据驱动的灵活性
4. **显式时间延迟建模**：延迟队列 + 历史窗口输入，解决纯滞后的时间相位对齐问题
5. **只优化 OPEX 不碰 CAPEX**：正确识别了 RTO 时间尺度下的可控成本
6. **传感器噪声防护层**：中值滤波 + EMA 平滑，禁止原始值直接重置预测起点
7. **DO 哨兵异常追踪**：利用 DO 异常升高间接检测不可测干扰（菌群中毒等）
8. **三级安全防护**（置信度监测 → 增量限幅 → 回退机制）：工程化落地的必要条件

## 单次 RTO 循环时间预算

| 步骤                         | 耗时         | 说明                |
| -------------------------- | ---------- | ----------------- |
| 采集当前工况数据（SCADA → RTO）      | ~0.1s      | 数据接口读取            |
| 传感器数据滤波（中值 + EMA）          | ~0.05s     | 去噪 + 平滑           |
| 异常检测（OOD + DO 哨兵）          | ~0.05s     | 分布外检测             |
| Layer 1 推理（ODE 积分预测）       | ~0.2s      | 刚性求解器             |
| Layer 2 优化求解（50-100 次梯度迭代） | ~0.5s      | 端到端梯度法            |
| Layer 3 约束检查、增量限幅          | ~0.1s      | 安全校验              |
| 输出最优控制指令 → 下发 PLC          | ~0.1s      | 通信接口              |
| **总计**                     | **~1-2 秒** | **远小于 15 分钟优化周期** |
